# Vibe Coding 方法手册
## 一、Vibe Coding 的核心认知

Vibe Coding 不是“让 AI 自动写代码”，而是“人主导，AI 执行”。

如果只把 AI 当代码生成器，结果通常是：

- 能跑，但结构混乱
- 功能能出，但质量不稳
- 前期很快，后期返工严重
- 一旦会话失控，就会陷入幻觉和死循环

教程里 12 篇经验技巧和 FAQ 反复指向同一件事：

1. 先规划，再执行
2. 先做 MVP，再迭代
3. 先给清晰上下文，再提需求
4. 让 AI 生成，也让 AI 验证
5. 及时把经验沉淀成规则、模板、Skill

## 二、五大基础心法

### 1. Planning is Everything

复杂任务不要上来就让 AI 开干。

更稳的做法：

- 先让 AI 反问需求，补齐目标、边界、约束
- 先产出 PRD、方案、文件改动计划
- 确认方向后再进入编码阶段

关键结论：`花 5 分钟规划，往往能省 30 分钟返工`。

### 2. MVP 思维

不要一开始追求完整系统。

更好的节奏：

- 先做一个最小可用版本
- 只实现 2 到 3 个核心功能
- 先验证方向，再补增强功能

### 3. 迭代优于完美

不要期待 AI 一次就给你完美答案。

更有效的方式：

1. 先生成骨架
2. 再补核心流程
3. 再修边界情况
4. 最后再做性能、体验和视觉优化

### 4. 上下文是王道

AI 很多时候不是不会写，而是不知道项目真实约束。

如果缺少上下文，AI 会自动选择最常见、最保守、最“像训练数据平均值”的方案。

### 5. 像产品经理一样思考

你不能只说“做个网站”或“加个功能”。

你需要明确：

- 用户是谁
- 核心场景是什么
- 功能优先级如何
- 体验细节要求是什么
- 哪些是必须，哪些可以后做

## 三、需求和对话工程

这部分对应教程里的“对话工程技巧”和 FAQ 中大量提问误区。

### 1. 好提示词的组成

一次高质量开发请求至少要包含：

- 项目背景
- 技术栈
- 本次目标
- 修改范围
- 约束条件
- 验证要求

参考模板：

```markdown
项目背景：
- 这是一个 xxx 项目，使用 xxx 技术栈。

本次目标：
- 实现 xxx 功能。

范围：
- 只允许修改 xxx 目录。

约束：
- 保持现有风格
- 不引入新依赖
- 所有新增函数补注释
- 保持 TypeScript 类型完整

验证要求：
- 跑测试
- 检查 lint / type error
- 总结风险点
```

### 2. 对话中的高价值动作

- 先让 AI 复述需求，确认理解一致
- 先让 AI 给计划，再决定是否执行
- 先做最小版本，再逐步增强
- 要求 AI 说明“为什么这么做”
- 要求 AI 给验证步骤，而不是只吐代码

### 3. 常见错误提法

低质量提法：

- “帮我做个系统”
- “修一下这个 bug”
- “做得现代一点”
- “优化一下”

高质量提法：

- “实现订单列表页筛选功能，只改前端，不改后端接口”
- “这个 bug 只在删除后二次刷新时出现，附完整报错和复现步骤”
- “页面风格不要蓝紫渐变，使用深灰 + 荧光绿，参考黑客终端氛围”

## 四、上下文工程

这是整套方法里最重要的一层。

### 1. 上下文三层模型

- 项目级：技术栈、目录结构、规范、设计风格
- 功能级：当前模块目标、边界、依赖关系
- 对话级：当前会话里临时讨论的信息

### 2. 必备长期记忆文件

建议至少维护：

- `README.md`：项目简介、技术栈、目录结构、运行方式
- `TODO.md`：已完成、进行中、待开发、已知问题
- `CLAUDE.md` / `.cursorrules`：AI 行为约束
- `/docs`：API、设计规范、技术说明、模块文档

### 3. 管理原则

- 相关即可，不是越多越好
- 信息必须及时更新，过时比缺失更危险
- 每个大功能结束后，把结果回写到文档
- 开新会话时，优先提供核心文档，不要重新空聊

### 4. FAQ 里高频问题的本质

常见问题比如：

- AI 忘了之前说的技术栈
- 代码风格前后不一致
- 前几轮还正常，后面开始乱写

本质往往不是模型太弱，而是：

- 对话太长
- 上下文被污染
- 项目规范没沉淀
- 当前任务和历史任务混在一起了

## 五、幻觉、死循环和排障

### 1. 幻觉的典型表现

- 编造不存在的 API
- 混用不同框架语法
- 使用旧版本写法
- 代码看着合理但跑不通
- 一直坚持错误实现

### 2. 死循环识别信号

满足任一条就要警惕：

- 连续 3 次方案本质一样
- AI 只是换写法，没有解决根因
- 会话过长，开始混淆目标
- 你自己都说不清当前状态

### 3. 正确止损方法

不要在原会话里硬耗。

推荐做法：

1. 总结目标
2. 列出已尝试方案
3. 提供最小复现和完整报错
4. 开新会话
5. 明确要求“换一种思路”

### 4. 正确喂报错信息

不要说“报错了修一下”。

至少要提供：

- 完整错误栈
- 相关代码片段
- 复现步骤
- 预期结果
- 实际结果

### 5. 判断问题到底在哪

教程和 FAQ 都强调过一个判断方法：

如果换成人类开发者，你这个需求本身都可能说不清，那问题未必在 AI，而在需求本身。

所以遇到问题要区分：

- 是 AI 的语法或实现错误
- 还是需求、流程、边界没有定义清楚

## 六、效率提升方法

这部分对应“效率提升技巧”和 FAQ 中关于速度、额度、上下文长度的内容。

### 1. 高效工作流

- 复杂任务先规划，再编码
- 任务按阶段拆会话
- 常用命令、常用规则写入配置文件
- 重复动作尽量脚本化或模板化

### 2. 命令行和自动化

教程特别强调 CLI 工具的价值：

- 启动快
- 适合远程开发
- 适合自动化
- 更适合 AI 调用

适合交给 CLI + AI 的任务：

- 批量处理文件
- 根据 `git diff` 生成提交信息
- 跑测试并分析失败原因
- 自动补注释、改文档、做检查

### 3. 会话收尾动作

每次任务结束后，建议让 AI 做 4 件事：

1. 总结改动
2. 列风险点
3. 给验证结果
4. 更新 TODO / 文档 / 规则文件

这一步是把“临时对话”变成“长期资产”的关键。

## 七、代码质量保障

这部分对应“代码质量保障”和 FAQ 中“AI 代码能不能直接用”的问题。

### 1. AI 代码常见质量问题

- 只追求能跑，不考虑结构
- 重复代码多
- 嵌套深
- 命名随意
- 错误处理缺失
- 边界情况未覆盖
- 注释和类型不完整

### 2. 质量保障最小清单

每次改动后至少检查：

- 是否通过编译
- 是否通过 lint
- 是否通过测试
- 是否有明显重复逻辑
- 是否有清晰命名
- 是否有必要的类型和注释
- 是否破坏现有功能

### 3. 高质量 Vibe Coding 的原则

- 小步提交，频繁验证
- 用 TypeScript 尽早暴露问题
- 让 AI 写测试，但人工检查测试是否真覆盖核心路径
- 把“为什么这样做”写进注释，而不是只写“做了什么”

## 八、重构与技术债

这部分对应“代码重构技巧”。

### 1. 技术债为何在 AI 项目里更严重

因为 AI 天然偏向：

- 优先让功能先跑起来
- 倾向在旧代码上补丁式叠加
- 不会主动长期维护架构一致性

### 2. 常见技术债信号

- 一个组件/文件过大
- 一个函数承担太多职责
- 多处复制粘贴
- 不敢改代码，怕牵一发而动全身
- 一加功能就引入连锁 bug

### 3. 重构原则

- 一次只改一小块
- 先保功能不变
- 先提取重复逻辑
- 先拆文件和职责，再谈高级抽象
- 每次重构都要有回归验证

### 4. 推荐重构顺序

1. 整理目录
2. 提取重复代码
3. 拆分大文件
4. 补类型定义
5. 改命名
6. 加测试和文档

## 九、性能优化

这部分对应“性能优化技巧”和 FAQ 里“项目越来越卡怎么办”。

### 1. 性能优化的基本态度

不要一开始就做过度优化。

推荐顺序：

1. 先保证功能正确
2. 再找真实瓶颈
3. 再做针对性优化

### 2. AI 容易给出的性能问题

- 不必要的重复渲染
- 无脑全量请求
- 大列表直接全渲染
- 图片资源过大
- 频繁计算没有缓存
- 前后端职责划分不合理

### 3. 可执行优化方向

- 大列表用分页/虚拟滚动
- 高频输入用防抖/节流
- 图片压缩、懒加载
- 只请求必要数据
- 减少不必要的状态和副作用
- 让 AI 先分析瓶颈再动手改

### 4. FAQ 里的常见误区

- 还没测量就盲目上 `useMemo`
- 后端慢却一直改前端
- 数据规模很小却过度复杂化

结论：`性能优化要基于瓶颈，不要基于想象。`

## 十、安全防护

这部分对应“安全防护技巧”和 FAQ 里关于密钥、环境变量、前后端职责的高频问题。

### 1. 最大风险：API Key 泄露

最常见错误：

- 把 Key 直接写进代码
- 把 `.env` 提交到 Git
- 在前端直接调用敏感第三方 API

### 2. 最小安全规则

- 所有密钥都走环境变量
- `.env` 必须加入 `.gitignore`
- 提供 `.env.example`
- 敏感调用放后端，不放前端
- 不要相信前端验证，后端必须再校验一次

### 3. 其他高频安全问题

- SQL 注入：必须参数化查询
- XSS：避免危险 HTML 注入
- 权限控制缺失：接口不能只靠前端拦
- 错误信息泄露：生产环境别回显敏感细节
- 依赖漏洞：定期 `npm audit`

### 4. 发布前安全检查

至少检查：

- 是否暴露密钥
- 是否有未鉴权接口
- 是否有不安全的用户输入处理
- 是否记录了敏感日志
- 是否做了基础备份和监控

## 十一、成本控制

这部分对应“成本控制技巧”和 FAQ 中关于额度、模型选择、长对话成本的内容。

### 1. 成本主要来自哪里

- 输入 token
- 输出 token
- 长上下文重复发送
- 使用过贵模型做低价值任务

### 2. 成本控制原则

- 简单任务别用最贵模型
- 输出比输入更贵，能少说就少说
- 长任务拆会话，别无限续聊
- 只引用必要文件，不要整库乱塞
- 稳定规则文件，利用缓存

### 3. 模型分工思路

- 便宜模型：写注释、文档、简单重构、测试数据
- 中档模型：常规功能开发、普通 bug 修复、代码审查
- 高档模型：复杂架构、疑难排障、大规模重构

### 4. FAQ 里的现实建议

- 不是什么都值得用 AI 做
- 手工 1 分钟能做完的，不要浪费 5 分钟和 token 去聊
- 预算和投入产出比要一起看

## 十二、团队协作

这部分对应“团队协作技巧”。

### 1. 团队里最重要的不是共享 AI，而是共享规则

如果没有统一规范，团队协作会出现：

- 风格不一致
- 技术栈混搭
- 改动冲突频繁
- 代码质量不可控

### 2. 团队协作最小制度

建议至少有：

- `README.md`
- `CONTRIBUTING.md`
- `CODE_STYLE.md`
- `API.md`
- `CHANGELOG.md`
- 统一 AI 规则文件

### 3. 必须统一的内容

- 技术栈
- 文件结构
- 命名规则
- 注释规范
- 提交规范
- PR 流程
- 代码审查重点

### 4. AI 在团队中的正确角色

- 不是替代协作机制
- 是加速规范执行、审查、文档更新、知识共享
- 团队应该共同维护规则文件和 Skill，而不是各自为战

## 十三、网站美化与去 AI 味儿

这部分对应“网站美化技巧”，也是 FAQ 里经常出现的“为什么页面一眼 AI”的根源。

### 1. AI 味儿的典型表现

- 蓝紫渐变泛滥
- 三栏卡片模板化
- 字体保守
- Emoji 滥用
- 内容空洞
- 缺少真实氛围和视觉主张

### 2. 本质原因

AI 为了求稳，会优先选择训练数据里最常见的“现代网页解法”。

所以如果你只说“做个现代网站”，它大概率就会产出平均值界面。

### 3. 去 AI 味儿的方法

- 让 AI 参考真实优秀网站
- 先做设计 Demo，再做完整开发
- 用明确的风格约束，不要泛泛而谈
- 用截图、参考站、文案语境注入氛围
- 使用更有风格的组件库，而不是默认大众方案
- 手工指定配色、字体、布局气质

### 4. 有效提示方式

低质量：

- “做个现代风官网”

高质量：

- “深灰背景、荧光绿强调色、终端感、拒绝蓝紫渐变、布局不对称、参考黑客控制台视觉”

### 5. 对前端 Skill 的启发

这类页面风格要求非常适合沉淀为前端设计 Skill：

- 品牌调性
- 色板
- 字体限制
- 布局约束
- 动效风格
- 禁止项

## 十四、FAQ 重组后的高频问题

### 1. 为什么 AI 前面写得好，后面越来越差？

通常是上下文过长、目标漂移、规范缺失，不一定是模型突然变弱。

### 2. 为什么 AI 总改坏别的地方？

通常是任务范围太大、一次改动太多、会话历史混乱。

### 3. 为什么同样的提示词效果不稳定？

因为上下文不同、文件不同、历史对话不同、模型不同。提示词不是唯一变量。

### 4. 为什么 AI 代码能跑但不好维护？

因为 AI 默认优先满足“当前能运行”，不会天然优先满足长期架构质量。

### 5. 什么时候该重开会话？

- 连续 2 到 3 次修不动
- AI 开始混淆技术栈
- 同一个问题越修越乱
- 你自己都描述不清状态时

### 6. 为什么页面总是一股 AI 味儿？

因为你给的是“平均化需求”，AI 返回的是“平均化答案”。

### 7. 为什么花费越来越高？

因为长对话、过多上下文、输出冗长、模型选择过度。

### 8. Skill 到底值不值得做？

非常值得。尤其是重复任务、固定流程、明确标准、团队共识很强的场景。

## 十五、最适合沉淀成 Skill 的知识

结合这 12 篇经验技巧和 FAQ，我认为优先级最高的 Skill 包括：

- `planning-skill`：需求澄清、PRD、任务拆解、方案评估
- `prompt-skill`：标准提问模板、范围控制、验证要求
- `context-skill`：README/TODO/规则文件维护方式
- `bugfix-skill`：最小复现、报错采集、死循环止损
- `quality-skill`：类型、lint、测试、注释、回归检查
- `refactor-skill`：技术债识别、拆分步骤、验证顺序
- `performance-skill`：定位瓶颈、选择优化手段、避免过度优化
- `security-skill`：密钥管理、输入校验、接口权限、发布前检查
- `cost-control-skill`：模型选择、上下文压缩、会话拆分、预算控制
- `team-collab-skill`：共享规范、PR 流程、审查要点、知识沉淀
- `frontend-style-skill`：去 AI 味儿、设计约束、参考站点、组件库选择

## 十六、推荐执行流程

如果要把整套教程真正落到实践，我建议用这个顺序：

1. 先建立 `README.md`、`TODO.md`、规则文件
2. 复杂任务先规划，形成文档
3. 每个会话只做一个主目标
4. 每步实现后立刻验证
5. 发现幻觉或死循环就及时止损
6. 每次完成任务后更新文档
7. 高频动作逐步提炼成 Skill
8. 团队场景下统一共享这些 Skill 和规则

## 十七、最终结论

这套经验技巧和 FAQ 最核心的价值，不在于教你“怎么把 prompt 写得更花”，而在于建立一套稳定的 AI 协作方法：

- 用规划减少返工
- 用上下文减少跑偏
- 用小步迭代减少失控
- 用验证保障质量
- 用重构控制技术债
- 用安全和成本意识避免后期事故
- 用 Skill 把一次次纠错变成长期复利

真正成熟的 Vibe Coding，不是“AI 替你写代码”，而是：

- 你知道该让 AI 做什么
- 你知道什么时候该打断它
- 你知道哪些经验值得沉淀成制度
- 你能让项目、团队和 AI 一起越来越稳
