# Skills 学习笔记

## 1. Skill是什么

Skill 是一组模块化、可复用、可触发的能力说明。它通常以一个独立目录存在，核心文件是 `SKILL.md`，也可以搭配脚本、参考资料和模板资源一起使用。

可以把它理解为：把某一类任务的经验、约束、流程和资源，整理成另一个 agent 可以直接使用的“执行手册”。它不是单纯的知识堆积，而是面向执行的能力封装。

skill 的结构如下：

```text
my-skill/
├── SKILL.md
├── agents/
│   └── openai.yaml
├── scripts/
├── references/
└── assets/
```

其中各部分职责不同：

- `SKILL.md`：核心入口，定义这个 skill 是什么、何时触发、怎么做
- `scripts/`：放确定性强、容易重复编写、适合脚本化的操作
- `references/`：放不应常驻上下文，但在执行中需要查阅的资料
- `assets/`：放模板、样例、图片、素材、样板工程等输出资源
- `agents/openai.yaml`：主要面向 UI 展示和技能元数据

## 2. Skill 的特点与价值

### 2.1 模块化与可触发

skill 是独立能力单元，有清晰边界、固定结构，也能基于 `name` 和 `description` 在合适场景被调用。

这让它既容易维护和扩展，也能减少重复解释，避免每次都从零组织 prompt。

### 2.2 分层加载与面向执行

skill 不是一份大而全的说明文档，而是按层加载、面向执行的能力载体：平时只暴露元数据，触发后读取 `SKILL.md`，需要时再加载 `references/`、`scripts/` 或 `assets/`。

这让它既能控制上下文成本，又能把重点放在“先做什么、再做什么、如何判断分支、如何验证结果”上，从而降低理解偏差和执行漂移。

### 2.3 可复用与可演进

一份写得好的 skill，不是服务一次请求，而是服务一类请求。它可以把脚本、资料、模板和经验沉淀下来，在相似任务中反复复用，并随着真实案例和失败复盘持续迭代。

这使 skill 不只是临时提示，而是可长期积累的能力资产。

## 3. 如何写好一个 Skill

写好 skill 的关键，不是写得多，而是写得准。

### 3.1 先理解真实使用场景

不要一上来写目录结构，先收集真实用例。至少要回答：

- 用户会怎么提需求
- 哪些请求应该触发这个 skill
- 哪些请求不该触发
- 执行过程里哪些地方最容易出错

### 3.2 优先写好触发描述

一个 skill 是否能被正确用到，首先取决于 `SKILL.md` frontmatter 里的 `description`。

好的 `description` 要同时回答三件事：

- 这个 skill 做什么
- 在什么场景下使用
- 典型输入、文件类型或任务类型是什么

差的例子：

```yaml
description: Help with documents
```

好的例子：

```yaml
description: Create, edit, and analyze .docx files with formatting preservation, tracked changes, comments, and content restructuring. Use when Codex needs to generate or modify Word documents, including reports, templates, formal letters, and review-marked files.
```

原则是：触发描述要具体、可识别、覆盖真实请求。

### 3.3 默认模型已经很聪明

不要把常识写进 skill。只写以下内容：

- 模型默认不知道的流程
- 特别容易做错的约束
- 团队特有的规则
- 需要读取哪个 reference
- 需要运行哪个 script
- 如何验证结果是否达标

可以用一句话判断是否该保留：

如果删掉这段话，Codex 仍然大概率能做对，那它通常不值得占用上下文。

### 3.4 控制自由度

skill 不是越细越好，而是要根据任务脆弱程度决定约束强度。

适合高自由度的情况：

- 有多种做法都合理
- 需要根据上下文判断
- 任务更偏分析和写作

适合中等自由度的情况：

- 存在推荐模式
- 允许一定变化
- 需要给出步骤和选择规则

适合低自由度的情况：

- 一旦做错代价高
- 顺序敏感
- 必须按固定方式执行
- 最好由脚本保证一致性

简化理解：

- 开阔地带，用指南
- 狭窄桥面，用护栏

### 3.5 用渐进式披露设计内容

不要把所有细节都塞进 `SKILL.md`。合理分层：

1. 元数据层：`name` + `description`
2. 主体层：`SKILL.md`
3. 资源层：`scripts/`、`references/`、`assets/`

推荐做法：

- `SKILL.md` 只写工作流、导航、约束、验证
- 大块知识放 `references/`
- 重复操作放 `scripts/`
- 输出模板放 `assets/`

这样可以让 skill 既轻量，又能覆盖复杂任务。

### 3.6 正文写成执行指南，不要写成宣传材料

好的 `SKILL.md` 应该告诉另一个 agent：

- 先做什么
- 后做什么
- 什么时候读哪个文件
- 哪种情况走哪条分支
- 用什么方式验证

不应该写成：

- 这个 skill 很强大
- 支持很多能力
- 适用于各种复杂任务

那种内容对执行几乎没有帮助。

### 3.7 让资源分工明确

判断是否需要资源文件，可以用这个简单规则：

- 代码每次都要重写：放 `scripts/`
- 知识经常要查、但不该常驻上下文：放 `references/`
- 最终输出要直接复用模板或素材：放 `assets/`

### 3.8 写清验证方式

一个 skill 如果没有验证方式，就很难稳定迭代。验证要求应尽量具体，例如：

- 是否成功生成目标文件
- 是否能被目标工具正常打开
- 是否通过格式校验
- 是否满足行文、结构或字段要求
- 脚本是否在代表性样例上运行成功

### 3.9 用真实任务驱动迭代

skill 的质量不是“看起来合理”，而是“在真实任务中可稳定复用”。最有效的方法不是空想，而是拿真实请求反复测试。

## 4. 如何评估 Skill

评估一个 skill，重点不是“内容多不多”，而是“能不能被正确触发，并且稳定完成任务”。

可以从以下 6 个维度评估。

### 4.1 触发准确率

看它是否会在正确场景触发，以及不该触发时不会误触发。

评估问题：

- 用户这样说时，会不会触发这个 skill
- 明显属于别的 skill 的请求，会不会被它误抢
- `description` 是否覆盖了典型表达方式

### 4.2 执行成功率

看 agent 使用这个 skill 后，是否能完成目标任务。

评估问题：

- 第一次执行能否顺利完成
- 是否经常卡在某一步
- 是否需要额外补充说明才能继续

### 4.3 路径清晰度

看另一个 agent 是否知道下一步该做什么。

评估问题：

- `SKILL.md` 是否清楚指向脚本、参考资料和输出路径
- 遇到分支情况时，是否有明确选择逻辑
- 是否存在大段“看了也不知道怎么用”的说明

### 4.4 上下文效率

看 skill 是否尽量少占上下文，又保留关键约束。

评估问题：

- `SKILL.md` 是否过长
- 是否把应放在 `references/` 的内容堆进正文
- 是否存在重复表述

### 4.5 输出稳定性

看多次执行是否结果接近，是否容易漂移。

评估问题：

- 不同任务样例下，结果是否保持一致标准
- 是否会因为表述变化而走偏
- 是否存在高频错误模式

### 4.6 可维护性

看 skill 是否易改、易扩展、易定位问题。

评估问题：

- 结构是否清晰
- 资源是否按职责拆分
- 修改某一类场景时，是否能局部更新

### 4.7 常用评估方法

可以按下面几种方式做评估：

- 静态检查：校验 frontmatter、命名和基础结构
- 样例测试：用 3 到 5 个真实请求测试触发和执行
- 对照测试：比较“有 skill”和“没 skill”的表现差异
- 前向测试：让独立 agent 按真实用户方式使用 skill
- 失败复盘：记录失败任务，反推 skill 的缺口

如果技能是新建的，至少应跑一次基础校验：

```bash
scripts/quick_validate.py <path/to/skill-folder>
```

## 5. 如何调优 Skill

调优 skill 的核心，不是“写更多内容”，而是“减少误解、减少浪费、提升命中和稳定性”。

### 5.1 先定位问题类型

skill 表现差，一般有 5 种常见原因：

- 触发不准：`description` 写得太泛或太窄
- 路径不清：正文没有明确先后顺序
- 信息过载：把太多细节塞进 `SKILL.md`
- 资源缺失：该有脚本/参考资料却没有
- 验证不足：做完了但不知道是否达标

先分型，再优化，不要盲目加内容。

### 5.2 优先优化 description

很多 skill 的问题根源不在正文，而在触发层。优化重点包括：

- 加入更明确的任务词
- 加入典型文件类型
- 加入排他性场景提示
- 补足常见用户表达方式

如果一个 skill 经常误触发，通常说明它写得太泛；如果经常漏触发，通常说明它覆盖不全。

### 5.3 缩短正文，增强导航

如果 `SKILL.md` 很长但效果不好，优先做两件事：

- 删除模型本来就知道的内容
- 把细节迁移到 `references/`

保留下来的内容应主要是：

- 工作流
- 分支选择
- 高风险约束
- 验证方式
- 资源入口

### 5.4 把重复动作脚本化

如果发现 agent 反复手写同类代码，或者同一步总容易出错，就应该把它改成脚本。

脚本化的收益通常有三点：

- 结果更确定
- token 更省
- 迭代更集中

### 5.5 针对失败案例反向补强

每次失败都值得沉淀成改进点。可以按这个思路复盘：

1. 失败发生在哪一步
2. 是触发问题、路径问题、资源问题还是验证问题
3. 应该改 `description`、`SKILL.md`，还是新增资源
4. 修改后是否能用同类任务复测通过

### 5.6 用前向测试检验泛化能力

真正可靠的调优，不是你自己看着顺眼，而是让另一个 agent 在尽量少背景的情况下去完成任务。

前向测试时应注意：

- 用真实任务表达，不要泄露答案
- 给最少必要上下文
- 不要把你的诊断结论提前告诉测试 agent
- 每轮测试后重新从原始材料出发判断结果

### 5.7 警惕“越改越重”

skill 很容易在迭代中膨胀。常见坏味道包括：

- 失败一次就加一大段说明
- 什么都往 `SKILL.md` 塞
- 参考文档和正文重复
- 出现大量低价值示例

调优目标不是变厚，而是变准。

## 6. 如何管理 Skill

skill 一旦多起来，管理成本会明显上升。管理重点不是“收集更多 skill”，而是保证它们可发现、可维护、可演进。

### 6.1 命名规范

skill 名称应满足：

- 使用小写字母、数字、连字符
- 语义清晰
- 尽量短
- 最好体现动作或领域

例如：

- `pdf`
- `skill-creator`
- `channel-access`
- `lark-calendar`

不要使用含糊、抽象、容易重名的名字。

### 6.2 目录管理

每个 skill 都应是独立目录，名称与 skill 名一致。推荐放在统一可发现的位置，例如：

- `$CODEX_HOME/skills`
- `~/.codex/skills`

如果是项目内专用 skill，也可以放在仓库下的 `.codex/skills/`。

### 6.3 结构管理

保持统一结构，方便理解和维护：

- 核心说明放 `SKILL.md`
- 脚本放 `scripts/`
- 参考资料放 `references/`
- 素材放 `assets/`

不要额外堆很多 README、CHANGELOG、说明文档，避免结构噪音。

### 6.4 生命周期管理

每个 skill 都应视为一个长期资产，至少经过以下生命周期：

1. 识别重复需求
2. 建立 skill
3. 在真实任务中使用
4. 根据失败案例迭代
5. 定期清理过时内容

如果一个 skill 长期没人用、触发混乱、资源失效，就应该修复、合并或淘汰。

### 6.5 版本与更新管理

虽然 skill 通常不强调显式版本号，但仍要管理变更质量。重点是：

- 更新后重新校验
- 更新后用代表性任务复测
- 检查 `agents/openai.yaml` 是否仍与 `SKILL.md` 匹配
- 确保引用的脚本、模板、路径没有失效

### 6.6 质量台账

建议至少维护这些观察维度：

- 哪些 skill 使用频率高
- 哪些 skill 经常失败
- 哪些 skill 容易误触发
- 哪些 skill 的资源已经过时
- 哪些 skill 存在功能重叠

如果不做这类盘点，skill 数量一多，质量会迅速失控。

### 6.7 避免重复建设

新建 skill 之前先问三个问题：

- 这个能力是否已有 skill 覆盖
- 是应该扩展旧 skill，还是新建 skill
- 多个相似 skill 是否会互相抢触发

有时候问题不该靠“再建一个 skill”解决，而应该靠重构已有 skill。

## 一套实用的方法论

如果要把 skill 真正写好并长期维护，可以记住下面这套闭环：

1. 从真实请求中发现重复模式
2. 把高价值流程抽出来
3. 用最小上下文写成 skill
4. 用脚本、参考资料、素材分层承载
5. 通过真实任务评估命中率和成功率
6. 根据失败案例持续调优
7. 用统一结构和规则长期管理

## 总结

Skill 的本质，不是“给模型补课”，而是把反复出现、容易出错、需要团队约束的工作方法沉淀为可触发、可执行、可验证、可复用的能力单元。

判断一个 skill 是否写得好，可以看 4 件事：

- 能不能在正确场景被触发
- 能不能指导另一个 agent 顺利完成任务
- 能不能少占上下文却保留关键约束
- 能不能在真实任务中持续迭代，而不是越写越臃肿

如果这 4 点都成立，skill 才真正从“文档”变成“能力”。
